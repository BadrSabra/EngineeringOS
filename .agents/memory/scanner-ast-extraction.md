---
name: Scanner AST rewrite
description: graph-extractor.ts extraction approach (TS/JS AST + Python via subprocess) and remaining coverage gaps, for anyone extending entity/relationship extraction.
---

- `lib/scanner/src/graph-extractor.ts` parses TS/JS via the TypeScript compiler API (`ts.createSourceFile`, `setParentNodes: false` since the walk never reads `.parent`) instead of line-by-line regex.
- CommonJS `require`/`module.exports`/`exports.foo =`, TS `export =`, and class methods as qualified function entities (`Class.method`) are **already implemented** in `extractFromTsJs` — an earlier version of this note called them deferred; that was stale. Remaining CommonJS gaps (`Object.assign(module.exports, {...})` batch exports, `module.exports = require(...)` re-exports) were closed too.
- Python is now parsed with **real structural parsing**, not regex: `extractGraph` batches every Python file in a scan into one `python3` subprocess (see `lib/scanner/src/python-extractor.ts`) that runs the interpreter's own `ast` module and returns per-file functions/classes/imports as JSON over stdout.
  - **Why a subprocess instead of an npm AST parser:** no lightweight, well-maintained JS/TS package parses Python to a full AST; shelling out to the real interpreter's stdlib `ast` module gives 100% grammar-accurate parsing for free. Batching (one process, all files via stdin) keeps subprocess-spawn overhead O(1) per scan instead of O(files).
  - Python was installed as a proper Replit module (`python-base-3.11`) rather than relying on an ambient system `python3` binary, so it's guaranteed present in production too — check `.replit`'s `modules` list before assuming any interpreter is available there.
  - The Python script is embedded as a generated TS string constant (`python-ast-script.ts`, built from the readable source `python-ast-script.py`) rather than read from disk at runtime — api-server bundles this package into a single esbuild output file, so a sibling `.py` asset would not be copied along, and any path relative to `import.meta.url` would resolve against the bundled file, not the original source location. **Regenerate `python-ast-script.ts` by hand after editing `python-ast-script.py`** (no build step wires this up automatically yet).
  - If the subprocess is unavailable, crashes, or a specific file has a syntax error, extraction falls back per-file to the old regex heuristic (`extractFromPythonRegex`) rather than failing the whole scan.
- `typescript` had to move from `devDependencies` to `dependencies` in `lib/scanner/package.json` once TS parsing became a runtime concern, not just a build-time one.
- `extractGraph` is `async` (because of the batched Python subprocess call) — every caller must `await` it; this rippled into `scan-runner.ts`, `discovery.ts`, and all scanner tests.
- **Scope-consistency bug pattern (fixed 2026-07-13):** the Python side used `ast.walk(tree)` (recurses into *every* nesting depth) to find classes, but only `tree.body` (module level only) to find functions. That mismatch let classes defined inside a function body leak into the graph as top-level entities (e.g. a local `class LocalHelper` inside `def outer():`), and a private-by-convention class's methods still surfaced as orphaned `_PrivateClass.method` entities even though the class itself was excluded. Fixed by a single recursive `collect_class_entities` helper that only descends into class bodies (nested classes like an inner `Meta` are still captured) and never into function bodies. When one AST walk uses a recursive traversal (`ast.walk`/similar) and a sibling walk uses a shallow one (`tree.body`), check whether their effective scope actually matches — it's an easy silent-inconsistency bug to introduce in either the TS or Python extractor.
- [Knowledge-engine BFS off-by-one](knowledge-engine-bfs-depth.md) — `getImpactedEntities`'s `maxDepthReached` counted a failed final hop; masked for a long time because the workspace-wide test run always aborted earlier in a sibling package.
