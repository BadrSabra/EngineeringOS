هذه ليست مجرد مجموعة تعديلات، بل **إعادة فصل مسؤوليات طبقة الاكتشاف (Discovery Layer)** إلى طبقات مستقلة. حاليًا يبدو أن `DiscoverProjectWizard` يعرف أكثر مما ينبغي عن طريقة إدخال المشروع، بينما المفترض أن يكون مجرد عميل (Client) لطبقة اكتشاف عامة.

 تقسيم التنفيذ إلى سبع مراحل، بحيث تكون كل مرحلة قابلة للاختبار والدمج دون كسر ما هو موجود.

# المرحلة السادسة — Source Adapters

هذه أهم طبقة ناقصة بالكامل تقريباً.

إنشاء

```
src/discovery/sources
```

بداخلها

```
BaseSourceAdapter.ts

LocalFolderAdapter.ts

WorkspaceAdapter.ts

GitRepositoryAdapter.ts

ArchiveAdapter.ts

RemoteFilesystemAdapter.ts

DockerVolumeAdapter.ts
```

كل Adapter ينفذ نفس الواجهة.

```
interface SourceAdapter {

validate()

prepare()

resolve()

cleanup()

metadata()

}
```

بعد ذلك

Discovery لا يعرف Git.

ولا يعرف Zip.

ولا يعرف Local.

بل يتعامل فقط مع

```
SourceAdapter
```

## خطوة إضافية أوصي بها

أثناء مراجعة تصور المشروع، أرى أن طبقة **Source Adapters** ينبغي ألا تكون مرتبطة مباشرة بطبقة **Discovery**، بل تصبح خدمة مستقلة قابلة لإعادة الاستخدام مثل:

```
lib/source-runtime/
├── adapters/
├── contracts/
├── resolver/
├── credentials/
├── workspace/
└── capabilities/
```

بهذا تصبح أي ميزة مستقبلية—مثل إعادة الفحص (Rescan)، المزامنة الدورية (Sync)، أو استيراد الإضافات (Plugin Import)—تعتمد على نفس طبقة المصادر دون تكرار الكود، ويصبح الاكتشاف مجرد أحد المستهلكين لها، وهو تصميم أكثر قابلية للتوسع ويتماشى مع الطبيعة متعددة الطبقات لمشروع EngineeringOS.
