

# PR 1 — Hardening & Contract Alignment for `projects.ts`

## الهدف

تحويل طبقة `projects.ts` إلى نقطة تحكم صلبة وموثوقة لإدارة المشاريع، بحيث تصبح:

* متسقة مع نموذج الصلاحيات الحالي.
* محصنة ضد اختلافات الحالات بين قاعدة البيانات والواجهة.
* واضحة في قواعد الإدخال/الإخراج.
* قابلة للاختبار على مستوى السلوك وليس فقط الترجمة.

## الملفات

* `artifacts/api-server/src/routes/projects.ts`
* `artifacts/api-server/src/lib/requireProjectAccess.ts`
* `artifacts/api-server/src/lib/requireAuth.ts`
* `lib/api-spec` أو أي ملفات توليد/عقود مرتبطة بحقول المشروع إذا وُجدت إشارات مباشرة من `projects.ts`
* اختبارات المسار المرتبطة بالمشروعات إن كانت موجودة في `artifacts/api-server/src/routes/__tests__` أو ما يماثلها

## التغييرات المطلوبة

1. **توحيد عقد بيانات المشروع**

   * مراجعة كل request/response shape في `projects.ts`.
   * تثبيت الحقول الإلزامية والاختيارية بشكل صريح.
   * منع أي اعتماد ضمني على payload غير موثق.
   * جعل رسائل الخطأ متسقة وقابلة للتمييز بين:

     * unauthorized
     * forbidden
     * not found
     * validation error

2. **تثبيت الصلاحيات**

   * التأكد أن كل endpoint حساس يمر عبر `requireAuth`.
   * التأكد أن أي operation على `projectId` يمر عبر `requireProjectAccess` أو تحقق مكافئ.
   * منع أي مسار يسمح بالوصول إلى مشروع عبر معرف فقط دون ملكية/صلاحية واضحة.

3. **توحيد التحقق من صحة المدخلات**

   * إدخال schema validation صريح لكل body/query/params.
   * إزالة أي parsing مباشر من `req.body` أو `req.query` دون تحقق.
   * جعل الرسائل المرجعية للخطأ قصيرة، ثابتة، ومصنفة.

4. **تثبيت قواعد الحالة**

   * منع انتقالات حالة غير متوافقة للمشروع.
   * توضيح متى يكون المشروع:

     * draft
     * active
     * archived
     * imported
     * pending initialization
   * إذا كان هناك lifecycle state حقيقي في الكود، يجب أن يُقفل في هذا الملف ولا يبقى موزعًا بين السطور.

5. **ربط العمليات بالـ audit/event model**

   * أي create/update/delete/status change يجب أن يكتب أثرًا واضحًا.
   * التأكد أن العمليات لا تغيّر البيانات فقط بل تترك trace يمكن مراجعته.
   * الحفاظ على correlation context إذا كان موجودًا في الخادم.

6. **تخفيف coupling**

   * فصل منطق الإجراء عن تعريف الراوتر إذا كان الملف يخلط بين routing وbusiness rules.
   * استخراج helpers محلية فقط إذا كانت تخدم readability أو testability.
   * منع تضخم الملف بغير ضرورة، لكن بدون تفكيك مبكر يضعف القراءة.

## الاختبارات

يجب إضافة أو تحديث الاختبارات التالية:

* إنشاء مشروع ناجح.
* منع إنشاء مشروع ببيانات ناقصة أو غير صالحة.
* منع الوصول لمشروع لا يملكه المستخدم.
* تحديث مشروع مع صلاحيات صحيحة.
* رفض أي تحديث غير مصرح به.
* مسارات not found / forbidden / validation error بشكل منفصل.
* أي منطق حالة state transition إن كان موجودًا.

## المخاطر

* كسر توافق الواجهة إذا كانت تعتمد على response shape غير ثابت.
* ظهور اختلاف بين validation الجديد والسلوك الحالي في بعض الحالات الحدّية.
* تضارب مع طبقة الصلاحيات إذا كان `projects.ts` يتجاوزها في أي مكان حاليًا.
* احتمال كشف اعتماديات قديمة على حقول لم تعد مناسبة.

## خطة rollback

* إعادة الملف `projects.ts` إلى النسخة السابقة.
* إبقاء schema التحقق الجديد خلف feature-safe edit إن أمكن.
* عدم تغيير contract الخارجي إلا إذا كان مدعومًا باختبارات تغطي المسارات الأساسية.
* إذا فشل أي test حرج، يتم إيقاف الدمج قبل لمس الملفات التالية.

## تعريف الاكتمال

يُعتبر هذا الـ PR مكتملًا فقط إذا تحقق الآتي:

* جميع مسارات `projects.ts` تمر عبر تحقق واضح.
* لا يوجد access غير مضبوط إلى project data.
* رسائل الأخطاء متسقة.
* الاختبارات الأساسية تمر.
* لا يوجد سلوك متناقض بين الصلاحيات والحالات والـ responses.
