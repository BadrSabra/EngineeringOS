## 1. Executive Summary

I scanned the whole extracted tree (`779` files) and traced the high-impact runtime paths. The codebase is organized around one central HTTP API server, a file scanner, a knowledge-graph/query layer, and an AI orchestration surface; the dashboard consumes that API, but it also contains a small amount of handwritten fetch logic that sits outside the generated client surface (`artifacts/dashboard/src/pages/AiChat.tsx:82-106`, `artifacts/dashboard/src/components/GitPanel.tsx:33-49`, `lib/api-client-react/src/use-ai-chat-stream.ts:2-7, 74-170`).

The strongest confirmed runtime truth is the boot-and-recovery chain: server startup runs `reconcileStuckJobs()`, then `fixDeadRootPaths()`, then starts listening, and only then schedules stale-job sweeping (`artifacts/api-server/src/index.ts:10-37`, `artifacts/api-server/src/lib/job-reconciliation.ts:283-294`, `artifacts/api-server/src/lib/startup-migrations.ts:29-60`). That means the server is explicitly designed to repair interrupted scans/discovery sessions before accepting traffic.

The most important state-changing flows are complete and well-instrumented: project discovery (`routes/discovery.ts:134-223, 284-380`), project scan (`routes/projects.ts:184-223`, `lib/scan-runner.ts:51-468`), AI chat/apply changes (`routes/ai.ts:486-670, 945-1060`), AI task execution (`routes/ai.ts:1352-1515`), and workflow orchestration (`routes/ai.ts:1222-1347`). These flows all write events/audit records and invalidate context cache where needed.

The largest confirmed gaps are operational rather than structural: scan/discovery jobs are backed by an in-process bounded queue (`lib/job-queue.ts:1-114`) and per-project LLM rate limiting is also in-memory (`routes/ai.ts:48-83`), so both degrade across restarts or multi-instance deployments. The root-path fallback logic also deliberately broadens AI file access to a workspace fallback when a stored project root disappears (`lib/rootpath-validator.ts:39-103`), which is safe only if that behavior is intended.

Documentation is mostly aligned with code, but not perfectly. The strongest mismatch is that `artifacts/api-server/src/services/task-service.ts` claims to contain “pure verification logic” while it actually reads from the database (`task-service.ts:1-69`). Another gap is the dashboard/client story: docs emphasize generated hooks, but the UI also uses direct `fetch()` for AI and Git operations (`AiChat.tsx:82-106`, `GitPanel.tsx:33-49`). I also verified that the source dependency boundary is respected in code: no `lib/*` source file imports `artifacts/*` runtime code.

## 4. Detailed Layer-by-Layer Analysis

### Presentation

The dashboard is not purely generated-client driven. It uses `@workspace/api-client-react` for many queries and the SSE hook, but it also has direct same-origin `fetch()` helpers for AI and Git endpoints (`AiChat.tsx:82-106`, `GitPanel.tsx:33-49`). That means the generated client is important but not exclusive.

### HTTP/API Transport

`app.ts` hardens the API surface with `etag` disabled, helmet, rate limiting, JSON/body size caps, CORS with credentials, and a centralized error handler (`app.ts:21-138`). Route-level auth is layered via `requireAuth` and `requireProjectAccess` (`requireAuth.ts:22-45`, `requireProjectAccess.ts:31-135`).

### Orchestration

The strongest orchestration code lives in `routes/ai.ts`, `scan-runner.ts`, `discovery-runner.ts`, and `job-reconciliation.ts`. These are the places where state mutates, jobs are queued, failures are recovered, and events/audits are emitted. The queue is process-local and bounded at 2 concurrent jobs (`job-queue.ts:110-114`).

### Domain/Verification

`workflow-service.ts` is genuinely pure: it only evaluates conditions and computes phase transitions (`workflow-service.ts:34-93`). `task-service.ts` is not pure despite the header comment; it reads `rulesTable` from the DB before deciding verification outcomes (`task-service.ts:1-69`).

### Scan/Extraction

`lib/scanner` is a real extraction pipeline, not a stub. It walks filesystem trees with file and size caps (`file-walker.ts:4-15, 151-190`), matches rules, extracts graph nodes/edges, and computes metrics (`lib/scanner/src/index.ts:1-21`).

### Persistence

`lib/db/src/index.ts` throws immediately if `DATABASE_URL` is absent and instantiates the shared `Pool` and `drizzle()` client (`lib/db/src/index.ts:1-16`). That makes DB availability a bootstrap dependency rather than a lazy runtime dependency.

### Contract Generation

`lib/api-spec/openapi.yaml` has 64 paths, and the route code exposes 64 unique path literals. That parity is confirmed by code inspection of route literals and the OpenAPI file.

## 9. Technical Debt Report

The debt is concentrated in coordination and completeness, not in basic CRUD plumbing.

1. Durable async work is not externalized. Both scans and discovery are queued in-process (`job-queue.ts:1-114`), then repaired at startup via DB reconciliation (`job-reconciliation.ts:1-294`). That is honest and workable, but it is still a crash-sensitive design.

2. Several safety features are per-process. Rate limiting is in-memory (`ai.ts:48-83`) and therefore only reliable on a single instance.

3. Discovery and scanning are bounded by hard caps rather than adaptive backpressure. That prevents OOM, but it also means the system can under-report very large or deeply nested repositories (`file-walker.ts:4-15, 94-145`).

4. The AI surface is operationally strong, but fragmented: generated clients, raw fetch, SSE hook, advisory locks, and cache invalidation all coexist. It works, but it raises maintenance complexity (`AiChat.tsx:82-106`, `GitPanel.tsx:33-49`, `use-ai-chat-stream.ts:2-7, 74-170`).
