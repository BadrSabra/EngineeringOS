# PR Title

**fix(discovery): Classify Project Loading Failures by Root Cause Instead of Reporting Generic Session Expiration**

---

# Problem Statement

الواجهة تعرض دائمًا:

> `Could not load your projects — your session may have expired.`

بينما الكود الحالي يعامل جميع حالات فشل `GET /api/projects` بنفس الطريقة.

هذا يجعل:

* Unauthorized
* Forbidden
* Internal Server Error
* Network Failure
* API Unavailable

تظهر جميعها كأنها Session Expired، وهو تشخيص غير صحيح ويصعب عملية التحقيق.

---

# Scope

يشمل هذا الـ PR فقط طبقة تحميل المشاريع.

لا يغير:

* Discovery Engine
* Scanner
* Project Registration
* AI Orchestration

## Phase 5

### Improve Error Mapping

بدلاً من

```
Could not load your projects.
```

استخدم:

| HTTP    | UI                                                  |
| ------- | --------------------------------------------------- |
| 401     | Session expired. Please sign in again.              |
| 403     | You don't have permission to access these projects. |
| 404     | Projects endpoint unavailable.                      |
| 500     | Internal server error.                              |
| Network | Cannot reach server.                                |

---

# Acceptance Criteria

* تصنيف جميع أخطاء تحميل المشاريع حسب السبب الحقيقي.
* عدم عرض رسالة "session expired" إلا عند رد `401` المؤكد.
* الاحتفاظ بمعلومات الخطأ (`status`، و`requestId` إن وجد، ونوع الفشل) حتى تصل إلى طبقة العرض.
* عدم كسر تدفق تحميل المشاريع عندما تكون الجلسة صالحة.
* تغطية اختبارات للحالات الأساسية: `200`، `401`، `403`، `500`، فشل الشبكة، والقائمة الفارغة.

# Rollback Plan

1. إعادة منطق عرض الرسالة القديم في `AiChat.tsx` إذا ظهرت مشكلات في واجهة المستخدم.
2. الإبقاء على تحسينات الـ logging والـ telemetry حتى عند التراجع.
3. التراجع عن تصنيف الأخطاء فقط دون تعديل `custom-fetch.ts` أو `requireAuth.ts` إذا ثبت أن السبب في طبقة العرض وحدها.
4. تنفيذ التراجع في Commit مستقل بحيث يمكن مقارنة السلوك قبل وبعد الإصلاح بسهولة.
