# EngineeringOS — Autonomous Project Discovery & Onboarding
# Execution Directive
# Version 1.0

## Mission

Do NOT improve the current "Register Project" form.

Instead, replace the entire registration workflow with a new autonomous onboarding experience that reflects the vision of EngineeringOS as an intelligent engineering operating system.

The objective is not to collect information from the user.

The objective is to let EngineeringOS discover, understand, validate, summarize and import the project automatically.

The user should feel that the system understands the repository before asking for any additional information.

This is a UX architecture task, not just a UI redesign.

--------------------------------------------------

# Phase 1
Repository Discovery Architecture

First perform a complete audit of the existing registration workflow.

Document:

• current UI flow
• current backend endpoints
• database interactions
• scanner entry point
• project creation lifecycle

Produce a sequence diagram before making any changes.

Do NOT modify code before understanding the current flow.

--------------------------------------------------

# Phase 2
Design a New Onboarding Flow

The new flow must replace

Register Project

with

Discover Project

The entire experience becomes:

Step 1
Choose Project Source

Supported sources:

• Local Workspace Folder
• Existing Workspace Repository
• Git Repository URL
• Uploaded Project Archive (future-ready)

The user should never need to type:

• Language
• Framework
• Runtime
• Package Manager

unless automatic detection completely fails.

--------------------------------------------------

# UX Principles

Never ask the user for information the system can discover.

Prefer discovery over forms.

Prefer confirmation over manual typing.

Reduce required user input to the absolute minimum.

# Scanner Integration

Reuse the existing scanner.

Do NOT duplicate scanner logic.

Do NOT create another discovery implementation.

The onboarding pipeline must become a thin orchestration layer over:

scanner

rule engine

metrics

graph extraction

repository analysis

--------------------------------------------------

# API Requirements

Create discovery endpoints if they do not exist.

Example flow:

POST /projects/discover

returns

Discovery Session ID

--------------------------------

GET /projects/discover/{id}

returns

Current Progress

--------------------------------

GET /projects/discover/{id}/summary

returns

Complete Discovery Report

--------------------------------

POST /projects/import

creates project from discovery result

without rescanning.

--------------------------------------------------

# Smart Detection Rules

Project name should come from:

package.json

Cargo.toml

go.mod

pom.xml

composer.json

pyproject.toml

or repository folder.

Language should be automatically inferred.

Framework should be automatically inferred.

Runtime should be automatically inferred.

Package manager should be automatically inferred.

Never ask the user first.

# Success Criteria

A first-time user should be able to onboard an existing repository by providing only its location.

EngineeringOS should discover everything else automatically.

The first interaction with EngineeringOS should demonstrate intelligence, not data entry.

The onboarding experience must communicate that EngineeringOS understands the repository before managing it.
