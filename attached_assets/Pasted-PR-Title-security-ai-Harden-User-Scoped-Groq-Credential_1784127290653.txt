PR Title

security(ai): Harden User-Scoped Groq Credential Management Before Merge

Summary

This PR performs a security verification and hardening pass over the recently introduced user-scoped Groq API key management.

The feature architecture is sound (per-user credentials, server-side injection, encrypted storage, dashboard management), but several security and operational requirements must be verified before the feature can be considered production-ready.

This PR does not introduce new functionality. Instead, it validates and hardens the implementation to ensure no credential leakage, incorrect encryption configuration, or unsafe fallback behavior exists.

Background

The previous implementation introduced:

User-scoped Groq API key storage
Encrypted credential persistence
Dashboard configuration UI
Per-user API key injection
OpenAPI endpoints
AI runtime integration

Before merging, a complete security verification is required.

Rollback Plan

If any security verification fails:

Disable user-scoped credential feature.
Revert runtime injection.
Keep Groq runtime on server default only.

Merge Gate

Do not approve this PR until all security verification items pass.

Mandatory blockers:

❌ Provider credential used as encryption key.
❌ Plaintext API key stored.
❌ Secret returned by API.
❌ Secret written to logs.
❌ Missing authentication on new endpoints.
❌ Unbounded Groq client creation.
