#1 - Set up the imported project
What & Why
The user just imported this project from a GitHub repository. It could be anything — an app to run on Replit, a library, a fork to study, code to reuse — so the first thing to do is ask what they want; everything after depends on the answer.

How
Take a quick look at the project (README, stack, entry points) — just enough to ask a good question. Don't review, verify, or change anything before the user answers. The .replit file here was generated by the import itself, so it says nothing about the user's intent.
Ask the user (AskQuestion) what they'd like: get it running on Replit, make specific changes, or leave it as imported for now. Mention anything you noticed that affects the answer (missing secrets, monorepo layout, nothing runnable, etc.).
If they want it running: install dependencies and configure the environment (ask for secrets or external services instead of inventing values), configure the run workflow, verify the app works (use screenshots for web apps), and record how to run it in replit.md. Keep the project's existing structure and stack — do not restructure or migrate it.
If they wanted something else, do that — don't set up a run workflow they didn't ask for.
Done looks like
The user got what they asked for and knows what, if anything, still needs their input.
Important
Make the smallest changes needed. If the project would benefit from bigger restructuring (e.g. migrating to Replit's pnpm workspace stack), propose that with proposeFollowUpTasks instead of doing it here.

Make archive-upload discovery survive a full host reboot without silent failure
What & Why
PR-D2 replaced the in-memory upload store with a DB-backed one. The happy path (upload → discover in one session) is tested. But the durability guarantee — that the entry survives a process restart and still resolves correctly — has no integration test. A regression here would silently give users a misleading "re-upload required" error mid-flow.

Done looks like
An integration test simulates the restart scenario: register an upload, wipe the in-memory state (restart context), then resolve the upload via the adapter and verify it returns the correct rootPath
A test verifies that if /tmp was wiped (access throws ENOENT), lookupUpload returns undefined and cleans up the DB row cleanly
These tests run in the existing vitest suite alongside discovery-adapters.test.ts
