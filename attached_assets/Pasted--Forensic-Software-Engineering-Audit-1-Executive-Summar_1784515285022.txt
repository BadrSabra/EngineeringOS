# Forensic Software Engineering Audit

## 1) Executive Summary

I audited the repository as a code-first system and treated docs as supporting evidence only. The repo is a pnpm workspace with **769 files**; the highest-concentration runtime code sits in `artifacts/api-server`, `lib/scanner`, `lib/ai-orchestrator`, `lib/knowledge-engine`, `lib/db`, and `artifacts/dashboard`. No `node_modules/`, `dist/`, `build/`, or `target/` trees were present in the extracted repo.

The most important confirmed findings are:

1. **Workflow condition evaluation is unsafe.**
   `artifacts/api-server/src/routes/workflows.ts:371-390` evaluates user-supplied `condition` strings with `new Function(...)`. The schema allows `phases[].condition` as a string (`lib/api-zod/src/generated/api.ts:722-730`). This is a server-side code execution risk in an authenticated path.

2. **The job system is process-local, not durable.**
   `artifacts/api-server/src/lib/job-queue.ts` is an in-memory queue, and `job-reconciliation.ts` exists to repair state after restarts. That means resilience depends on reconciliation rather than durable queue semantics.

3. **The core product flow is real, not just documented.**
   Discovery, scanning, graph extraction, task execution, workflow advancement, AI chat/apply, and audit/event persistence are all implemented across `discovery.ts`, `scan-runner.ts`, `tasks.ts`, `workflows.ts`, `ai.ts`, and the DB schema.

4. **The architecture is layered but not thin.**
   The API routes contain substantial business logic instead of acting as pure transports, especially `ai.ts`, `tasks.ts`, and `workflows.ts`.

5. **The repo is unusually documentation-heavy.**
   Docs, prompts, and supporting notes are numerous, but several backlog docs are stale relative to the code.

## 4) Detailed Layer-by-Layer Analysis

### L-001 Bootstrap / Configuration

* `artifacts/api-server/src/index.ts:10-17` starts reconciliation before listening.
* `artifacts/api-server/src/config.ts` enforces environment correctness instead of silently defaulting.
* `package.json` blocks npm/yarn usage and defines codegen/build/test workflows.

### L-002 HTTP Transport

* `artifacts/api-server/src/app.ts:31-124` mounts middleware in a conventional order: proxy trust, Clerk, helmet, rate limit, logging, CORS, JSON limit, routes, error handler.
* In test mode it bypasses external auth and swaps in synthetic auth logic.
* This layer is operationally complete, but it also hosts domain logic in route handlers.

### L-003 Orchestration / Jobs

* `job-queue.ts` is bounded and in-memory.
* `job-reconciliation.ts` repairs unfinished jobs and discovery sessions after restart.
* `scan-runner.ts` performs the full scan pipeline and persists graph/task/metric state transactionally.
* This is a real orchestration layer, but not durable like an external queue.

### L-004 AI Adapter / AI Orchestrator

* `lib/ai-orchestrator/src/context-builder.ts` builds an evidence-rich project snapshot using a repeatable-read transaction plus cache.
* `groq-client.ts` centralizes provider access with retries and a circuit breaker.
* `parsing.ts` hardens model-output parsing.
* The agent schemas and parse-failure markers are wired into routes so invalid output can return 422 instead of corrupting state.

### L-005 Scanner

* `file-walker.ts` imposes explicit file limits and skip rules.
* `graph-extractor.ts` extracts relationships and provenance from source code, with AST handling for TypeScript and a Python subprocess path.
* `metrics-calc.ts` computes structural metrics, but some outputs are heuristic rather than measured.

### L-006 Knowledge Engine

* `lib/knowledge-engine/src/index.ts` exports traversal, evidence, layered views, and summary functions.
* It is read-heavy and pure in intent, with no direct mutation path.
* This layer is well-bounded and comparatively mature.

### L-007 Domain State / Persistence

* `projects.ts`, `tasks.ts`, `workflows.ts`, `graph.ts`, `events.ts`, `metrics.ts`, `audit_logs.ts`, `scan_jobs.ts`, `ai_chat_*` define the true state surface.
* The model is provenance-aware and event/audit-oriented.
* Ownership appears single-user rather than multi-tenant org-based.

### L-008 Presentation

* Dashboard pages consume generated client hooks and render state, but do not define core logic.
* This layer is supportive and mostly complete enough for the currently exposed API surface.

### L-009 Generated Contracts

* Generated schema/client trees are derivative and should be treated as drift-prone artifacts.
* They are important, but not source-of-truth code.

## 9) Technical Debt Report

1. **Queue durability debt**
   Root cause: in-process queue.
   Impact: crash/restart recovery relies on reconciliation.
   Fix: durable queue or outbox pattern.

2. **Security debt in workflow expressions**
   Root cause: string evaluation with `new Function`.
   Impact: server-side code execution within authenticated scope.
   Fix: replace with safe expression parser or constrained AST interpreter.

3. **Metric semantics debt**
   Root cause: heuristic coverage labeled as coverage.
   Impact: false confidence in quality reporting.
   Fix: rename to `structuralEstimate` or integrate real coverage telemetry.

4. **Layering debt**
   Root cause: business logic in route files.
   Impact: testing and change isolation suffer.
   Fix: extract service/use-case modules.

5. **Generated drift debt**
   Root cause: generated contracts are stored in repo and may drift.
   Impact: API/client mismatch.
   Fix: enforce codegen drift checks in CI.

## 16) Final Engineering Verdict

The repository is **substantially implemented** and the core architecture is real: discovery, scanning, graph inference, task lifecycle, workflow transitions, AI orchestration, audit logging, and dashboard consumption all exist in code.

The main blocker is not incompleteness in the usual sense; it is **safety and operational hardness**:

* the workflow condition evaluator is dangerously permissive,
* the queue is not durable,
* and one quality metric is structural rather than measured.

So the system is **functional but not yet safe enough to trust as a production-grade execution kernel** until the top risks are repaired.
