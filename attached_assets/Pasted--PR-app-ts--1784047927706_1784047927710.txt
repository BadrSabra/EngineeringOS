فيما يلي خطة PR تنفيذية متسلسلة، ملفًا ملفًا، تبدأ من `app.ts` وتتحرك من طبقة الحماية إلى طبقة المشاريع ثم الاكتشاف ثم التنفيذ ثم الرصد والذكاء. الصياغة هنا عملية وموجهة مباشرة للتنفيذ.

---

## PR 1 — `app.ts`

**الهدف:** تثبيت هيكل التطبيق، ترتيب الـ middleware، وضمان أن كل مسار يمر عبر طبقة الحماية والتتبع بشكل متسق.

**الملفات:**

* `artifacts/api-server/src/app.ts`

**التغييرات:**

* تثبيت ترتيب middleware بشكل نهائي: `helmet` → `rateLimit` → `cors` → `clerkMiddleware` → `requireAuth`.
* إضافة Request ID موحّد لكل طلب وربطه بالـ logs والـ errors.
* توحيد التعامل مع أخطاء 404/500 في نقطة مركزية واحدة بدل تشتتها.
* التأكد من أن مسارات الـ health / readiness / debug لا تكسر نموذج الحماية.
* تنظيم route mounting بحيث تكون مسارات الـ API واضحة ومفصولة عن غيرها.

**الاختبارات:**

* اختبار أن الطلبات غير المصرّح بها تُرفض قبل الوصول إلى handlers.
* اختبار أن 404 و500 يعودان بصيغة موحدة.
* اختبار أن requestId يظهر في الاستجابات/اللوغز.
* اختبار عدم كسر أي route موجود بعد إعادة ترتيب الـ middleware.

**المخاطر:**

* كسر غير مقصود في ترتيب التنفيذ.
* ظهور اختلافات في سلوك بعض المسارات الخاصة إذا كانت تعتمد ضمنيًا على ترتيب قديم.

**rollback:**

* إعادة `app.ts` إلى الترتيب الحالي إن ظهر أي regression في auth أو routing.
* إبقاء التغييرات محصورة في هذا الملف فقط في هذا الـ PR.

## PR 2 — `requireAuth.ts` و`requireProjectAccess.ts`

**الهدف:** تحويل التحقق من الهوية والملكية إلى طبقة صلبة ومستخدمة على نطاق كامل.

**الملفات:**

* `artifacts/api-server/src/middleware/requireAuth.ts`
* `artifacts/api-server/src/middleware/requireProjectAccess.ts`

**التغييرات:**

* جعل `requireAuth` يثبت `userId` و`sessionId` وحقول الهوية بشكل موحد.
* جعل `requireProjectAccess` هو المرجع الوحيد للتحقق من ملكية المشروع.
* إضافة تفريق واضح بين:

  * authenticated
  * project member
  * project owner
  * project write access
* تحسين رسائل الخطأ لتكون قابلة للتتبع داخل logs.
* التأكد من أن أي مسار يمرر `projectId` لا يعتمد على الثقة بالمعرّف فقط.

**الاختبارات:**

* user authenticated بدون ملكية لا يصل إلى مشروع ليس له.
* user يملك المشروع يمر بنجاح.
* user لا يملك write access يفشل في PATCH/DELETE/scan.
* 401 و403 يعودان بشكل منفصل وصحيح.

**المخاطر:**

* اكتشاف مسارات كانت تعمل سابقًا بالاعتماد على ثغرة access ضمنية.
* احتمال ظهور مسارات ناقصة الحماية بعد تطبيق تشديد أمني.

**rollback:**

* إعادة logic التفويض إلى النسخة السابقة مؤقتًا مع إبقاء logs لتحديد المسارات المتأثرة.

## PR 3 — `projects.ts`

**الهدف:** تثبيت طبقة المشاريع كطبقة مرجعية آمنة ومنضبطة.

**الملفات:**

* `artifacts/api-server/src/routes/projects.ts`

**التغييرات:**

* الحفاظ على `ownerId` من `req.userId` فقط، ومنع أي override من body.
* فرض `requireProjectAccess` و`requireProjectWriteAccess` على كل endpoints الخاصة بالمشروع.
* توحيد response shape في list/get/create/update/delete/scan.
* التحقق الصارم من `projectId` قبل أي قراءة/تعديل.
* إبقاء scan enqueue فقط داخل route، وعدم تنفيذ العمل الثقيل داخل request cycle.

**الاختبارات:**

* إنشاء مشروع يربطه تلقائيًا بـ owner الصحيح.
* قراءة/تعديل/حذف مشروع من مستخدم غير مالك يفشل.
* scan route ينشئ job فقط ولا ينجز scan داخل الطلب.
* قائمة المشاريع تعرض فقط مشاريع المستخدم الحالي.

**المخاطر:**

* أي endpoint كان يعتمد على permissive access سيبدأ بالفشل، وهذا مقصود لكنه يحتاج رصدًا.
* احتمال وجود client code قديم يعتمد على response shape غير موحد.

**rollback:**

* التراجع عن strict guards فقط عند الضرورة القصوى، مع الاحتفاظ بخيار logging للحالات المرفوضة.

## PR 4 — `discovery.ts`

**الهدف:** تثبيت onboarding الاكتشافي كمحرك فعلي لفهم المشروع وليس مجرد import endpoint.

**الملفات:**

* `artifacts/api-server/src/routes/discovery.ts`
* `artifacts/api-server/src/lib/discovery-adapters.ts` إذا كان متصلاً مباشرة بسير العمل

**التغييرات:**

* توحيد اكتشاف المصدر: local path / repo / workspace source.
* تشديد التحقق من root path ووجوده قبل enqueue أو import.
* ضمان أن discovery session تُستهلك مرة واحدة فقط.
* جعل create project + metrics + graph seed + tasks + events داخل transaction قدر الإمكان.
* تنظيم فرق واضح بين:

  * resolve source
  * scan source tree
  * detect stack/features
  * persist discovery output
  * enqueue follow-up work
* حفظ سبب الفشل التفصيلي في discovery session بدل رسالة عامة.

**الاختبارات:**

* source صالح ينتج session منتهية بنجاح.
* root path مفقود أو غير صالح يفشل مبكرًا.
* إعادة import لنفس المصدر لا تكرر نفس الكيان بلا ضوابط.
* failure في الاستيراد لا يترك بيانات جزئية غير متسقة.

**المخاطر:**

* discovery معقد بطبيعته، وأي تعديل في sequencing قد يسبب duplicate entities أو partially imported projects.
* التغييرات هنا قد تكشف اعتمادًا قديمًا على سلوك غير محدد.

**rollback:**

* التراجع إلى path سابق محفوظ لعملية discovery مع إبقاء checks الجديدة على source validation.

## PR 5 — `scan-runner.ts`

**الهدف:** تقوية pipeline الفحص ليصبح قابلًا للتتبع والاستعادة والاختبار.

**الملفات:**

* `artifacts/api-server/src/lib/scan-runner.ts`

**التغييرات:**

* تثبيت lifecycle واضح للـ scan: start → walk → match → extract → persist → complete/fail.
* جعل `correlationId` إلزاميًا ومكتوبًا في كل event/metric/audit متصل بالفحص.
* التأكد أن فشل scan يعيد المشروع إلى حالة سليمة بدل تركه عالقًا.
* إحكام التعامل مع الملفات الكبيرة أو المسارات العميقة.
* تقليل الاعتماد على side effects غير الموثقة أثناء التحليل.

**الاختبارات:**

* scan ناجح يحدّث الحالة والمقاييس والـ graph.
* scan يفشل في منتصفه ثم يترك المشروع بحالة صالحة.
* correlationId يظهر في كل الأثر الناتج.
* حدود الحجم/العمق تمنع الانفجار.

**المخاطر:**

* احتمالية كسر بعض النتائج إذا كان هناك اعتماد ضمني على ordering قديم.
* أي bug هنا يظهر على شكل نتائج scan غير مكتملة أو مكررة.

**rollback:**

* إعادة lifecycle القديم مع الاحتفاظ بأكبر قدر من التحسينات في telemetry.

## PR 6 — `tasks.ts`

**الهدف:** جعل المهام مرتبطة بملكية المشروع والتتبع بشكل صارم.

**الملفات:**

* `artifacts/api-server/src/routes/tasks.ts`

**التغييرات:**

**المخاطر:**

**rollback:**

## PR 7 — `rules.ts`

* قد يظهر تعارض مع بيانات قديمة غير مرتبطة جيدًا بالمشروع.
